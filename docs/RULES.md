# Project Rules

## Commit conventions

- always use commitlint (Conventional Commits) pattern for commit messages
- commits must contain only the subject line â€” no body, no footer

## Design

- base UI primitives in `src/components/ui/` are generated by shadcn (Button, Input, Label, Alert, Field, Separator, etc.). Do not edit them unless strictly necessary â€” prefer composing them, or wrapping in a higher-level component in `src/components/`. When an edit is truly required, keep the shadcn shape (cva variants, `data-slot`, `cn(...)` order, `asChild` via `Slot.Root`) so future regenerations remain mergeable
- always render copy through `src/components/ui/text.tsx` (`<Text as="p" variant="â€¦">`) and `src/components/ui/heading.tsx` (`<Heading as="h1" variant="â€¦">`). Never reach for raw `<p>`, `<span>`, `<h1>`â€¦`<h6>`, etc. in feature code. This keeps typography (sizes, weights, tracking, muted tones) consistent across the app; pick the semantic `as` and lean on `variant`/`size` for tweaks
- custom composite components (those that wrap or combine `ui/` primitives into a higher-level reusable piece) live in `src/components/composite/`. Keep one component per file, kebab-case filenames. Reserve `src/components/ui/` for shadcn primitives only and `src/components/form/` for form-bound compositions

## Code patterns

- private (module-scoped, non-exported) functions prefix `_` â€” e.g. `_computePepper`, `_peppered`. Marks intent: not part of public API
- functional factory pattern: export `createX()` that returns a plain object literal of methods. No explicit return type â€” let TS infer the shape from the literal
- avoid declaring `type`/`interface`. Prefer implicit inference everywhere â€” return types, factory shapes, callback params. Only declare a type when truly necessary (public API contract consumed cross-module, recursive shape, or generic constraint). Inline param types on closures are fine
- meaningful variable names everywhere (prod code AND tests). No single-letter or filler names (`a`, `b`, `x`, `h`, `out`, `tmp`, `data`, `result`). Name by role: `firstDigest`/`secondDigest`, `hasher`, `originalSecret`, `childProcess`. Exception: idiomatic loop indices (`i`, `j` in nested numeric loops) and common functional callbacks (`(item, index) =>`)

## Database

- all SQLite database files (`dev.db`, `test.db`, `prod.db`) live in `db/` at the repo root. Never at project root or anywhere else
- `DATABASE_URL` in `.env*` always uses the form `file:./db/<name>.db` (e.g. `file:./db/dev.db`, `file:./db/test.db`)
- `db/` is committed via `.gitkeep`; the `.db` files themselves are git-ignored

## Test conventions

- tests use `bun:test` with `it("should â€¦")` phrasing â€” semantic BDD, not bare `test(...)`
- spec layout mirrors source: `src/lib/foo.ts` â†’ `test/lib/foo.spec.ts`
- single outer `describe` named after the module/export (e.g. `.hash`). Nested `describe` per exported method, prefixed with `.` (e.g. `.create`, `.compare`). Tests unrelated to a specific method live at the outer describe level, not inside a method block
- test env lives in `.env.test` at repo root (auto-loaded by `bun test` since `NODE_ENV=test`). Do NOT set `process.env.X` in `beforeAll` for static config â€” use the env file
- avoid subprocess (`Bun.spawn`) tests unless genuinely required (e.g. asserting behavior that depends on a fresh module evaluation / cleared module-level cache). When needed, keep it to a single `it`, not a whole describe
- imports go at the top of the spec file with static `import` syntax. Do NOT use per-test `await import(...)` unless you genuinely need to scope evaluation (e.g. asserting import-time side effects, or an inline import inside a subprocess `-e` string). ESM modules are cached by resolved path anyway â€” dynamic import per `it` gives the same instance, just more noise
- inside each `it`, separate **arrange / act / assert** with a single blank line. Cleanup in `finally` stays inside the `try/finally` block. Skip the separators only when the whole test is a single inline expression
